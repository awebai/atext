the trust model

Who says you are who you say?

Every day your team's agents act for you — write a doc, hand a stranger a link, call another company's service. Each time, the other side has to answer one question: who is this, and do they really speak for their team? The usual answer is “they have an account with us.” awid answers it with no account at all. Here is the story of how.

You already prove who you are every day

You own a domain. When someone loads acme.com or gets an email from it, they are already trusting that you — and only you — control it. awid turns that everyday fact into identity. You publish one small DNS record that says this key speaks for acme.com. Because no one but the domain's owner can edit its DNS, the record itself is the proof. There is no signup form and no company deciding to let you in: the root of trust is something you already hold.

see the actual record
dns
# published once, at the domain you control
_awid.acme.com  TXT  "awid=v1; controller=did:key:z6Mk…"
# add  registry=https://…  to run your own registry; otherwise api.awid.ai
# the record bootstraps the controller key; that key then signs the
# namespace's actions, and DNS stays the root authority for recovery
what about subdomains?
delegation
# a verified parent can authorize a child — no second DNS record
acme.com  (verified)  →  dev.acme.com   # child gets its own key
# the parent controller signs the child operation; not automatic inheritance

A key you can replace without becoming a stranger

That record names a key. But keys get lost, leaked, or simply rotated — so awid splits identity in two. Your stable id is who you are, and it never changes. Your signing key is what you sign with today, and you can swap it any time without becoming a stranger — like keeping your passport number while replacing a worn-out signature. Each change is appended to a signed log, so a verifier can tell an ordinary rotation from a suspicious one. Not every agent needs all this: one working inside a single workspace can just be a key and a team card. The ones that need to be found and followed across systems also get a public handle — an address like acme.com/alice — that points back to the stable id.

see how an identity is named
identity
did:aw:2CiZ88h…             # the stable identity — this is "you"
  ├─ did:key:z6MkehRg…    # current signing key; rotates
  └─ acme.com/alice           # a public handle bound to the identity

# a local, workspace-only agent is just a did:key + a team certificate —
# no did:aw, no address, and not recoverable if the key is lost
how a key change is judged
key log
# every key change is appended to a signed, hash-chained log
OK_VERIFIED   # signature + chain check out — a normal rotation
OK_DEGRADED   # usable, but falls back to trust-on-first-use
HARD_ERROR    # rewritten history — reject

A team is a group your domain vouches for

One identity is not a team. So the domain's controller creates one — backend:acme.com — with a key of its own. That team key then signs a certificate for each member: a small card that says this key belongs to this team. Because only the team key can sign it, no one can forge membership or quietly add themselves. A teammate can even live on another domain: partner.com/bob can hold a card for your team without leaving theirs.

look inside a certificate
certificate
{
  "team_id":        "backend:acme.com",
  "certificate_id": "…",
  "member_did_key": "did:key:z6Mk…",   # the key this card authorizes
  "alias":          "alice",
  "identity_scope": "global",               # or "local"
  "member_did_aw":  "did:aw:2CiZ88h…",  # global members only
  "member_address": "acme.com/alice",       # global members only
  "team_did_key":   "did:key:z6Mk…",   # convenience copy — verifiers
                                            #   fetch the real one from awid
  "issued_at":      "…Z",              # revoked at awid, never expires
  "signature":      "…"               # signed by the team key
}

Proving it, one request at a time

Showing the card is not enough — a card can be photocopied. So every time an agent acts, it signs the exact request: this action, on this path, carrying this body, meant for this app, at this moment. The app re-checks each of those against what actually arrived, traces the card back to the key your domain published, and confirms the member was not revoked. If a single thing is off — wrong audience, stale timestamp, unknown team — it refuses. It fails closed, never open.

see the bytes a request signs
request
# one signature covers all of this
{ "v":2, "method":"POST", "path":"/v1/documents",
  "body_sha256":"…", "aud":"https://api.atext.ai",
  "team_id":"backend:acme.com", "timestamp":"…Z" }

# the app verifies, in order — any miss fails closed
  1 canonical + v2     4 body hash matches      7 cert key == signer
  2 timestamp fresh    5 method/path/aud/team   8 team key from awid
  3 signature valid    6 …bound to request    9 not revoked

Why go to all this trouble?

Because it buys things accounts cannot. No passwords to leak or reset. Private keys never leave your machine — the registry only ever stores public facts. Revoke a member once and they are locked out everywhere, instantly. It spans organizations without anyone rehoming their identity. And every link traces back to something you already own and already control: your domain.
